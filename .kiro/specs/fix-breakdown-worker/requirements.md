# 修复 Breakdown Worker 执行逻辑

## 1. 概述

修复 `breakdown/start` 接口及其 worker 执行逻辑中的问题，确保任务提交后能够成功将章节内容发送给大模型进行拆解，并正确处理各种异常情况。

## 2. 问题描述

当前实现存在以下问题：

1. **配额回滚逻辑缺失**：任务失败时配额不会被返还
2. **安全漏洞**：使用 `eval()` 解析 JSON，存在代码注入风险
3. **错误处理不完善**：某些异常情况下批次状态不一致
4. **模型调用可能失败**：缺少对模型适配器调用的充分测试和错误处理

## 3. 用户故事

### 3.1 作为用户，我希望任务提交后能成功调用大模型
**验收标准：**
- 任务提交后，worker 能够正确加载章节数据
- worker 能够成功调用模型适配器的 `generate()` 方法
- 模型返回的结果能够被正确解析
- 拆解结果能够保存到数据库

### 3.2 作为用户，我希望任务失败时配额能被返还
**验收标准：**
- 任务失败时，已消耗的配额应该被返还给用户
- 配额返还操作应该在事务中完成，保证数据一致性
- 配额返还失败不应该阻止错误信息的记录

### 3.3 作为用户，我希望看到清晰的错误信息
**验收标准：**
- 模型调用失败时，应该记录详细的错误信息
- 错误信息应该包含失败原因、时间戳等关键信息
- 错误信息应该使用 JSON 格式存储，避免安全漏洞

### 3.4 作为用户，我希望实时看到大模型的拆解过程
**验收标准：**
- 前端 system console 组件能够实时显示大模型返回的流式信息
- 流式信息应该包含当前执行的步骤和模型输出
- 流式信息应该通过 WebSocket 实时推送到前端
- 用户能够看到每个拆解技能的执行进度和结果

### 3.5 作为开发者，我希望有完善的测试覆盖
**验收标准：**
- 单元测试覆盖所有核心函数
- 集成测试验证完整的拆解流程
- 测试应该包含成功和失败场景

## 4. 功能需求

### 4.1 配额回滚功能
- 实现同步版本的配额回滚逻辑
- 在任务失败时自动调用配额回滚
- 记录配额回滚操作的日志

### 4.2 安全的 JSON 解析
- 移除所有 `eval()` 调用
- 使用 `json.loads()` 安全地解析 JSON
- 添加异常处理，防止解析失败导致程序崩溃

### 4.3 增强的错误处理
- 捕获模型调用过程中的所有异常
- 区分可重试和不可重试的错误
- 记录详细的错误堆栈信息

### 4.4 模型调用验证
- 验证模型适配器是否正确初始化
- 验证 API Key 是否有效
- 验证模型返回的数据格式

### 4.5 实时流式输出
- 使用 WebSocket 推送实时日志到前端
- 支持流式调用大模型 API
- 将模型的流式响应实时发送到前端 console
- 记录每个拆解步骤的开始和结束时间

## 5. 非功能需求

### 5.1 性能
- 模型调用超时时间设置为 120 秒
- 数据库操作应该使用连接池
- 避免不必要的数据库查询

### 5.2 可靠性
- 使用事务保证数据一致性
- 实现自动重试机制（最多 3 次）
- 记录详细的执行日志

### 5.3 可维护性
- 代码结构清晰，职责分明
- 添加详细的注释和文档
- 使用类型提示提高代码可读性

## 6. 技术约束

- 使用同步数据库操作（Celery worker 环境）
- 兼容现有的模型适配器接口
- 不改变 API 接口的签名
- 保持向后兼容性

**P1 - 应该修复：**

## 10. 风险和缓解措施

### 风险 1：模型 API 调用失败
**缓解措施：**
- 实现重试机制
- 添加详细的错误日志
- 提供降级方案

### 风险 2：配额回滚失败
**缓解措施：**
- 使用数据库事务
- 记录回滚失败的日志
- 提供手动修复工具

### 风险 3：数据库连接问题
**缓解措施：**
- 使用连接池
- 设置合理的超时时间
- 实现自动重连机制
