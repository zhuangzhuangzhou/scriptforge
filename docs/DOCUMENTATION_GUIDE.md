# 文档编写指南 (Documentation Guide)

**版本**: v1.0.0  
**更新日期**: 2026-02-10  
**状态**: 📘 规范文档

---

## 🎯 目标

本指南为所有项目文档的编写提供统一标准，确保：
- 文档结构清晰、易于导航
- 命名规范统一、易于查找
- 内容分类明确、职责清晰
- 更新策略明确、避免混乱

---

## 📝 文档分类与职责

### 1. 快速开始文档 (`01-getting-started/`)

**目的**: 帮助新用户快速上手

**包含内容**:
- `quickstart.md`: 5-10 分钟快速开始
- `installation.md`: 详细安装步骤
- `configuration.md`: 配置说明

**更新频率**: 中等（功能变化时更新）

**命名规范**: `kebab-case.md`

---

### 2. 核心规范文档 (`02-specifications/`)

**目的**: 定义系统的核心规则和约定

**包含内容**:
- `workflow.md`: 系统流程规范
- `config-driven.md`: 配置驱动规范
- `storage-contract.md`: 数据与文件关系
- `document-policy.md`: 文档策略
- `document-versioning.md`: 文档版本规范

**更新频率**: 低（仅在规范变化时更新）

**更新原则**:
- ⚠️ 更新需谨慎，必须团队讨论
- ✅ 更新后需同步 `README.md` 和 `docs/index.md`
- ✅ 必须记录版本号和更新日期
- ✅ 必须在文档顶部标注状态：📘 规范文档

**命名规范**: `kebab-case.md`（不使用 `spec-` 前缀，通过目录区分）

- ✅ 归档文档保留原始内容，不删除

**命名规范**: 保持原命名

## 🏷️ 文档命名规范

### 通用规则

1. **统一使用小写 `kebab-case`**
   - ✅ 正确: `backend-guide.md`
   - ❌ 错误: `Backend_Guide.md`, `BackendGuide.md`

2. **命名清晰、可读，避免缩写**
   - ✅ 正确: `model-management-quickstart.md`
   - ❌ 错误: `mm-qs.md`

3. **不使用前缀区分类型（通过目录区分）**
   - ✅ 正确: `02-specifications/workflow.md`
   - ❌ 错误: `spec-workflow.md`（旧规范，需迁移）

### 特殊命名

1. **日期相关文档**: `YYYY-MM-DD-topic.md`
   - 示例: `2026-02-10-database-migration.md`

2. **README 文件**: 始终大写 `README.md`

3. **索引文件**: 使用 `index.md`

## 📊 文档状态标识

在文档顶部使用状态标识，帮助读者快速了解文档类型：

- 📘 **规范文档**: 核心规范，更新需谨慎
- ✅ **已完成**: 功能已完成，文档稳定
- 🚧 **进行中**: 功能开发中，文档可能变化
- 📝 **计划中**: 功能计划中，文档为草稿
- 🗄️ **已归档**: 历史文档，仅供参考
- ⚠️ **已废弃**: 不再使用，即将删除

## 🔄 文档更新流程

### 1. 更新规范文档

```bash
# 1. 确认需要更新规范
# 2. 与团队讨论变更
# 3. 更新文档内容
# 4. 更新版本号和日期
# 5. 在"更新记录"中添加条目
# 6. 同步更新 README.md 和 docs/index.md
# 7. 提交 PR 并审查
```

### 2. 更新功能文档

```bash
# 1. 更新文档内容
# 2. 更新日期
# 3. 如有重大变更，更新状态
# 4. 提交 PR
```

### 3. 添加修复记录

```bash
# 1. 在 docs/08-changelog/ 或相应位置创建文档
# 2. 使用修复记录模板
# 3. 问题解决后，移至 docs/09-archive/fixes/
```

## 🎨 文档编写最佳实践

### 1. 清晰的标题层级

```markdown
# 一级标题（文档标题，仅一个）
## 二级标题（主要章节）
### 三级标题（子章节）
#### 四级标题（细节说明）
```

### 2. 使用 Emoji 增强可读性

- 📖 概述
- ✨ 特性
- 🚀 快速开始
- 📚 文档
- 🔧 配置
- 🎯 目标
- ⚠️ 警告
- ✅ 完成
- ❌ 错误
- 💡 提示

### 3. 代码块使用语言标识

```markdown
\`\`\`python
# Python 代码
\`\`\`

\`\`\`typescript
// TypeScript 代码
\`\`\`

\`\`\`bash
# Shell 命令
\`\`\`
```

### 4. 使用表格组织信息

```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 值1 | 值2 | 值3 |
```

### 5. 使用引用块强调重要信息

```markdown
> **重要**: 这是一条重要信息。

> **注意**: 这是一条注意事项。

> **提示**: 这是一条提示信息。
```

### 6. 链接使用相对路径

```markdown
✅ 正确: [文档](../02-specifications/workflow.md)
❌ 错误: [文档](/docs/spec-workflow.md)
```

### 7. 保持文档简洁

- 一个文档专注一个主题
- 避免重复内容，使用链接引用
- 使用列表和表格提高可读性
- 代码示例要完整可运行

## 🤖 AI 协作指南

### 给 AI 的指令

当 AI 需要创建或更新文档时，应遵循以下规则：

1. **查看本指南**: 首先阅读 `docs/DOCUMENTATION_GUIDE.md`
2. **确定文档类型**: 根据内容选择正确的目录
3. **使用正确模板**: 根据文档类型使用对应模板
4. **遵循命名规范**: 使用 `kebab-case` 命名
5. **更新索引**: 如果添加新文档，更新 `docs/index.md`
6. **检查链接**: 确保所有链接有效

### 常见场景

**场景 1: 创建新功能文档**
```
1. 在 docs/05-features/ 下创建子目录
2. 使用功能文档模板
3. 更新 docs/index.md
```

**场景 2: 记录修复过程**
```
1. 在 docs/08-changelog/ 创建修复记录
2. 使用修复记录模板
3. 问题解决后移至 docs/09-archive/fixes/
```

**场景 3: 更新规范**
```
1. 确认是否真的需要更新规范
2. 更新 docs/02-specifications/ 中的文档
3. 更新版本号和日期
4. 同步更新 README.md 和 docs/index.md
```
