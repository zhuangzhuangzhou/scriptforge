# Trellis 开发工作流

## 核心原则

1. **先读后写** - 修改代码前必须先阅读相关文件
2. **遵循规范** - 按照 spec/ 目录中的规范开发
3. **任务驱动** - 使用任务系统跟踪开发进度
4. **质量检查** - 实现后必须进行代码检查

## 目录结构

```
.trellis/
├── workflow.md          # 本文件 - 工作流指南
├── spec/                # 开发规范
│   ├── frontend/        # 前端规范
│   ├── backend/         # 后端规范
│   └── guides/          # 通用指南
├── scripts/             # 工具脚本
└── tasks/               # 任务目录
```

## 开发流程

### 1. 理解任务
- 明确目标和需求
- 确定任务类型（前端/后端/全栈）

### 2. 研究代码库
- 查找相关规范
- 找到现有代码模式
- 确定需要修改的文件

### 3. 创建任务
- 使用 task.sh 创建任务目录
- 编写 prd.md 需求文档
- 配置上下文文件

### 4. 实现
- 调用 implement agent
- 遵循注入的规范

### 5. 检查
- 调用 check agent
- 运行 lint 和类型检查
- 修复发现的问题

### 6. 完成
- 验证所有检查通过
- 提交代码
- 归档任务

## 最佳实践

- 小步提交，频繁验证
- 保持代码风格一致
- 编写清晰的提交信息
- 及时更新文档
