# 小说转剧本业务流转规范 (Novel to Script Workflow)

## 1. 核心状态定义 (Status Lifecycle)

项目生命周期由以下 6 个核心状态组成，定义在 `Project.status` 字段：

| 状态 | 中文名 | 含义 | 触发条件 | 下一步 |
| :--- | :--- | :--- | :--- | :--- |
| **`draft`** | 草稿 | 项目已创建，仅包含名称、类型等元数据。 | 用户创建项目 | 上传源文件 |
| **`uploaded`** | 已上传 | 小说源文件已上传并存储 (MinIO)，但尚未处理。 | 文件上传成功 | 智能拆分章节 |
| **`ready`** | 已就绪 | **关键节点**。小说已成功解析并拆分为结构化章节 (`chapters` 表)。 | 调用 `/split` 接口成功 | 启动项目 (分批) |
| **`parsing`** | 拆解中 | 项目已启动，系统自动完成分批 (`Batch`)，正在进行剧情分析。 | 点击【项目启动】 | 剧情大纲生成 |
| **`scripting`** | 剧本中 | 剧情大纲确认，正在批量生成剧本 (`episodes` 表)。 | 触发剧本生成任务 | 剧本审核/导出 |
| **`completed`** | 已完成 | 所有批次剧本生成并通过审核。 | 所有任务完成 | 归档/导出 |

### 状态流转图

```mermaid
graph LR
    Start((开始)) --> draft
    draft -- 上传文件 --> uploaded
    uploaded -- 智能拆分 --> ready
    ready -- 项目启动(自动分批) --> parsing
    parsing -- 拆解完成 --> scripting
    scripting -- 全部生成 --> completed
```

## 2. 关键接口逻辑 (Key Logic)

### 2.1 章节拆分 (`POST /{id}/split`)
**目标**: 将非结构化的 TXT/DOCX 文件转化为结构化的 `Chapter` 记录。
- **输入**: 项目 ID。
- **前置检查**: 状态必须为 `uploaded` 或 `ready` (允许重试)。
- **核心逻辑**:
  1. 读取 MinIO 中的源文件。
  2. 使用 `ChapterSplitter` (正则/空行) 解析文本。
  3. **全量覆盖**: 删除旧章节，批量插入新章节。
  4. 更新统计数据 (`total_words`, `total_chapters`)。
  5. **状态更新**: `status` -> `ready`。

### 2.2 项目启动 (`POST /{id}/start`)
**目标**: 正式进入 AI 处理流程，完成预处理工作。
- **输入**: 项目 ID。
- **前置检查**: 状态必须为 `ready`。
- **核心逻辑**:
  1. **自动分批 (Batching)**: 调用 `BatchDivider`，根据项目设置的 `batch_size` (如每5章)，将所有章节划分为若干 `Batch`。
  2. **关联数据**: 创建 `Batch` 记录，并更新 `Chapter.batch_id`。
  3. **状态更新**: `status` -> `parsing`。
  4. (下一步): 前端自动跳转至 **PLOT (剧情拆解)** 页面，展示批次列表。

## 3. 前端交互规范 (Workspace UI)

### 3.1 布局结构
- **左右分栏**: 左侧 (68%) 展示配置与统计，右侧 (32%) 独立展示源文件管理。

### 3.2 右侧源文件面板 (Source Panel)
根据项目状态动态展示：
- **无文件**: 显示大尺寸虚线上传区域。
- **`uploaded`**: 显示文件信息，操作按钮为 **【开始智能拆分】**。
- **`ready`**: 显示"章节拆分完成"提示，引导点击顶部启动按钮。允许【重新拆分】/【重新上传】。
- **`parsing`+**: 显示"源文件已锁定"，禁止修改操作。

### 3.3 顶部操作栏
- **【项目启动】按钮**:
  - 仅在 `ready` 状态下**高亮激活**。
  - 点击触发 `startProject` 接口。
  - 成功后自动跳转至 `PLOT` 标签页。
