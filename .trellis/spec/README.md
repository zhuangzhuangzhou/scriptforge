## 🎯 如何使用规范

### 开发前
1. 阅读相关规范，了解约定和模式
2. 检查是否有类似的实现可以参考

### 开发中
1. 遵循规范，保持代码一致性
2. 遇到问题时查阅规范中的"常见错误"部分

### 开发后
1. 如果发现新的模式或问题，使用 `/trellis:update-spec` 更新规范
2. 在代码审查时参考规范

## 🔄 规范更新流程

当你学到新知识时：

1. **识别知识类型**
   - 新模式 → 添加到"最佳实践"
   - 常见错误 → 添加到"常见错误"
   - 陷阱 → 添加警告说明

2. **选择目标规范**
   - 安全相关 → `backend/security.md`
   - 数据库相关 → `backend/database.md`
   - API 设计 → `backend/api-design.md`

3. **更新规范**
   - 使用模板（见下方）
   - 包含代码示例
   - 解释"为什么"

4. **记录更新**
   - 在 `SPEC_UPDATE_LOG.md` 中记录

## 📋 更新模板

### 添加新模式

```markdown
### 模式名称

**问题**：描述要解决的问题

**解决方案**：简要说明

**示例**：
\`\`\`python
# ✅ 好的做法
code example

# ❌ 不好的做法
bad code example
\`\`\`

**为什么**：解释原因
```

### 添加常见错误

```markdown
### 常见错误：描述

**症状**：会出现什么问题

**原因**：为什么会发生

**修复**：如何解决

**预防**：如何避免
```

## 🎓 规范哲学

> **规范是活的文档。每次调试、每个"啊哈时刻"都是改进规范的机会。**

目标是**机构记忆**：
- 一个人学到的，所有人受益
- AI 在一次会话中学到的，持续到未来的会话
- 错误变成有文档记录的防护栏
