# 拆解任务问题诊断指南

## 问题描述
1. **WebSocket 连接失败** - 前端无法建立 WebSocket 连接，回退到轮询模式
2. **轮询显示重复消息** - 轮询模式一直显示相同的进度信息（如"识别场景"）

## 根本原因

### 问题 1: WebSocket 连接失败
可能的原因：
- Redis 服务未运行或连接失败
- WebSocket 端点配置错误
- 前端 WebSocket URL 不正确
- 防火墙或代理阻止 WebSocket 连接

### 问题 2: 轮询显示重复消息
**已修复但未生效**：代码已经更新为显示 "提取冲突中... (20%)" 等带进度的消息，但 **Celery worker 需要重启** 才能加载新代码。

## 解决方案

### 步骤 1: 检查 Redis 连接

Redis 是 WebSocket 实时推送的基础。检查 Redis 是否正常运行：

```bash
# 测试 Redis 连接（使用你的 SSH 隧道端口 6380）
redis-cli -h localhost -p 6380 ping
```

预期输出：`PONG`

如果失败，检查：
- SSH 隧道是否正常：`ps aux | grep ssh | grep 6380`
- Redis 服务器是否运行

### 步骤 2: 重启 Celery Worker（重要！）

**这是解决轮询重复消息问题的关键步骤。**

代码已经更新，但 Celery worker 还在运行旧代码。必须重启：

```bash
# 方法 1: 使用提供的脚本（推荐）
cd backend
./restart_celery.sh

# 方法 2: 手动重启
# 2.1 查找并停止现有 worker
ps aux | grep celery | grep worker
kill <进程ID>

# 2.2 启动新 worker
cd backend
source venv/bin/activate
celery -A app.core.celery_app worker --loglevel=info
```

### 步骤 3: 验证更新是否生效

重启 Celery worker 后，启动一个新的拆解任务，观察进度显示：

**预期看到的新格式：**
- ✅ "初始化任务中... (0%)"
- ✅ "加载章节数据中... (10%)"
- ✅ "提取冲突中... (20%)"
- ✅ "识别情节钩子中... (35%)"
- ✅ "分析角色中... (50%)"
- ✅ "识别场景中... (65%)"
- ✅ "提取情感中... (80%)"
- ✅ "保存拆解结果中... (90%)"
- ✅ "任务完成 (100%)"

**如果还是看到旧格式（如"识别场景"），说明 Celery worker 没有重启成功。**

### 步骤 4: 检查 WebSocket 端点

确认 WebSocket 端点已注册：

```bash
# 查看 FastAPI 路由
cd backend
source venv/bin/activate
python -c "from app.api.v1.router import api_router; print([r.path for r in api_router.routes if '/ws/' in r.path])"
```

预期输出应包含：
- `/ws/breakdown/{task_id}`
- `/ws/breakdown-logs/{task_id}`
- `/ws/batch-progress/{project_id}`

### 步骤 5: 检查前端 WebSocket URL

前端连接 WebSocket 时使用的 URL 格式应该是：

```javascript
// 对于单任务进度
ws://localhost:8000/api/v1/ws/breakdown/{task_id}

// 对于实时日志
ws://localhost:8000/api/v1/ws/breakdown-logs/{task_id}
```

**常见错误：**
- ❌ 使用 `http://` 而不是 `ws://`
- ❌ 缺少 `/api/v1` 前缀
- ❌ 端口号错误
- ❌ task_id 格式错误或任务已完成

## 测试 WebSocket 连接

可以使用以下工具测试 WebSocket 连接：

### 使用 wscat（需要安装）
```bash
# 安装 wscat
npm install -g wscat

# 测试连接（替换 task_id）
wscat -c "ws://localhost:8000/api/v1/ws/breakdown-logs/your-task-id"
```

### 使用浏览器控制台
```javascript
// 在浏览器控制台运行
const ws = new WebSocket('ws://localhost:8000/api/v1/ws/breakdown-logs/your-task-id');
ws.onopen = () => console.log('✅ WebSocket 连接成功');
ws.onerror = (e) => console.error('❌ WebSocket 连接失败:', e);
ws.onmessage = (e) => console.log('收到消息:', e.data);
```

## 常见问题排查

### Q1: Celery worker 重启后还是显示旧格式？
**A:** 确认以下几点：
1. 确实停止了所有旧的 worker 进程：`ps aux | grep celery`
2. 新 worker 启动时加载的是正确的代码路径
3. 没有使用缓存的 `.pyc` 文件：`find backend -name "*.pyc" -delete`

### Q2: WebSocket 连接一直失败？
**A:** 检查：
1. Redis 是否正常运行（见步骤 1）
2. FastAPI 服务器是否支持 WebSocket（应该默认支持）
3. 前端 URL 是否正确（见步骤 5）
4. 浏览器控制台是否有错误信息

### Q3: 轮询模式可以工作，但 WebSocket 不行？
**A:** 这说明：
- ✅ 后端 API 正常
- ✅ 数据库连接正常
- ❌ WebSocket 连接有问题（可能是 Redis 或网络配置）

建议：
1. 先确保 Redis 正常
2. 检查防火墙设置
3. 如果在生产环境，检查反向代理（如 Nginx）的 WebSocket 配置

## 验证修复

完成上述步骤后，验证修复是否成功：

1. **重启 Celery worker**（最重要！）
2. **启动一个新的拆解任务**
3. **观察前端进度显示**：
   - 如果看到 "提取冲突中... (20%)" 等新格式 → ✅ 修复成功
   - 如果还是 "识别场景" 等旧格式 → ❌ Celery worker 未重启或重启失败

4. **测试 WebSocket 连接**：
   - 打开浏览器开发者工具 → Network → WS 标签
   - 启动任务，观察是否有 WebSocket 连接建立
   - 如果有连接且收到消息 → ✅ WebSocket 正常
   - 如果没有连接或连接失败 → ❌ 需要检查 Redis 和网络配置
